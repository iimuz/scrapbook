Git で GPG を利用してアクセストークンの暗号化

2018-11-10 22:48:47 +0900 +0900 2020-02-19 23:34:31 +0000 UTC

下記の記事を参考にして GPG を利用して GitHub へ push するときに、 .netrc を利用してアクセストークンを入力するようにしました。

Qiita: gitレポジトリとhttpで通信する場合にユーザ情報の入力を省略する

  1. GPG で暗号化が可能な状態にする

  2. GitHub でアクセストークンを発行する

    • トークンの発行時に repo にだけチェックを入れておけば、 push はできるようです。
    • GitLab の場合は、最初の api の部分だけで push できました。

  3. .netrc を下記のように作成します

    machine github.com
login username
password access_token

  4. gpg で暗号化すると .netrc.gpg が作成される

    $ gpg -e -r hoge .netrc

  5. 暗号化した .netrc.gpg をホームディレクトリ下に配置する

  6. 最初に作成した .netrc は削除する

  7. git credential helper を設定する。 git-credentail-netrc は、 git 環境に付属しているようです。 また、環境によって微妙にインストールされている場所が異なるので、 それぞれの環境に合わせてください。

    $ chmod +x /usr/share/doc/git/contrib/credential/netrc/git-credential-netrc
$ git config --global credentail.helper /usr/share/doc/git/contrib/credential/netrc/git-credential-netrc

  8. 後は普通に git push

たまに下記のようなエラーが出るときがあります。

$ gpg -r hoge -d geho.gpg
gpg: encrypted with 3072-bit RSA key, ID XXXXXXXXXXXXXXXX, created 2018-11-10
      "hoge <hoge@example.com>"
      gpg: public key decryption failed: Permission denied
      gpg: decryption failed: No secret key

実際に .gnupg のディレクトリの実行権限が正しくない場合もありますが、 うまく tty でパスフレーズを確認する画面を表示できないことが原因の場合もあるようです。 その場合は、 using with su/sudo によると、 --pinentry-mode=loopback というフラグを使うと解消できます。 フラグを追加すると、パスフレーズを聞くような CUI の画面が新規にでるのではなく、 普通に 1 行だけ入力を促すように出力されます。

$ gpg --pinentry-mode=loopback -r hoge -d geho.gpg
Enter passphrase:

あと、 --pinentry-mode=loopback は最後に着けるのは正常に動作しません。 gpg コマンドはフラグの位置が重要っぽい挙動をするのでちょっと使いにくい時があります。

最初は、うまく解除ができないようです。 一回だけどこかで復号化などの処理を実施しておくと正常に実行できます。

$ gpg --pinentry-mode=loopback -r hoge -d geho.gpg
